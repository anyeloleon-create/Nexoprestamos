##  Nombre del Proyecto y Detalles

<div align="center">

# NexoPréstamos

### *El nexo entre tus objetos y las personas en las que confías*

</div>

**NexoPréstamos** es un sistema de gestión de préstamos desarrollado en Python orientado a consola, diseñado para ayudar a cualquier persona que preste artículos con frecuencia y necesite llevar un control claro, ordenado y automatizado de sus préstamos, devoluciones y facturación.

El nombre **NexoPréstamos** combina *nexo* (del latín *nexus*: vínculo, conexión, red) con la acción de *prestar*, transmitiendo la idea de que el sistema es el puente formal y confiable entre quien presta y quien recibe. Es moderno, profesional y comunica exactamente la propuesta de valor del sistema.

### ¿Qué hace NexoPréstamos?

```
🔹 Registra ítems       →   Con categoría, precio y estado difuso
🔹 Gestiona usuarios    →   Con validación completa de datos
🔹 Controla préstamos   →   Con fechas, alertas y recordatorios
🔹 Genera documentos    →   Certificados de devolución y facturas de venta
🔹 Reporta estadísticas →   Estado general de préstamos ordenados por días
🔹 Panel de admin       →   Acceso restringido con usuario y contraseña
```

##  Reporte de Visión

### Descripción General

**NexoPréstamos** es un software de consola desarrollado en Python que permite gestionar de manera eficiente el inventario de artículos prestados entre personas. El sistema nace como respuesta a una necesidad real: el olvido que tienen los seres humanos cuando se prestan objetos cotidianos como herramientas, videojuegos, libros y electrodomésticos sin un registro formal.

### Problema que Resuelve

En contextos informales de préstamos entre amigos o conocidos, la falta de registro genera:
- Pérdida de artículos sin posibilidad de reclamación
- Conflictos por desinformación sobre fechas y acuerdos
- Imposibilidad de hacer seguimiento a múltiples préstamos simultáneos
- Ausencia de mecanismos formales para la recuperación o cobro de artículos

### Objetivo del Software

Proporcionar una herramienta de consola intuitiva y confiable que permita:

1. **Registrar** usuarios e ítems con toda su información relevante
2. **Gestionar** préstamos con fechas, estados y condiciones acordadas
3. **Notificar** y alertar sobre préstamos próximos a vencer o ya vencidos
4. **Generar** documentos formales de devolución y facturación
5. **Administrar** el sistema con control de acceso y reportes ejecutivos

### Beneficios Clave

| Beneficio | Descripción |
|-----------|-------------|
|  **Ahorro de tiempo** | Consultas rápidas del estado de cualquier préstamo |
|  **Organización** | Toda la información almacenada en archivos planos estructurados |
|  **Protección económica** | Generación automática de facturas luego de 30 días |
|  **Alertas tempranas** | Notificaciones desde los 20 días de préstamo |
|  **Documentación formal** | Certificados y recibos que respaldan los acuerdos |
|  **Control de acceso** | Panel de administración protegido con credenciales |

### Usuarios Objetivo

- Personas que prestan artículos frecuentemente a amigos o familiares
- Pequeños colectivos que gestionan préstamos de herramientas o materiales
- Cualquier entorno donde se requiera un sistema liviano de control de préstamos


##  Especificación de Requisitos

### Requisitos Funcionales

Los requisitos funcionales definen qué debe hacer el sistema:

| ID | Requisito | Descripción |
|----|-----------|-------------|
| RF-01 | Registrar Usuario | El sistema debe permitir registrar usuarios con nombre, apellido, documento, correo electrónico y tiempo de préstamo (5, 10, 15 o 30 días), con validaciones completas en cada campo |
| RF-02 | Registrar Ítem | El sistema debe permitir registrar artículos con nombre, categoría (Videojuegos, Libros, Música y video, Herramientas, Dinero, Misceláneo), precio de compra, ID único por categoría y estado mediante lógica difusa |
| RF-03 | Registrar Préstamo | Solo se pueden crear préstamos a usuarios existentes. El sistema debe listar el inventario disponible y asociar el ítem al usuario seleccionado con registro de fecha |
| RF-04 | Registrar Devolución | Solo se permiten devoluciones de préstamos activos. Al devolver antes del plazo, se genera un certificado en archivo de texto plano |
| RF-05 | Generar Factura de Venta | Los ítems con más de 30 días prestados generan automáticamente una factura con subtotal, impuesto del 23% y total |
| RF-06 | Consultar Estado de Préstamos | El sistema muestra los ítems prestados ordenados por cantidad de días, leyendo la información de archivos planos |
| RF-07 | Módulo Administrador | Acceso restringido con usuario y contraseña. Reportes de: total préstamos, ítems devueltos, ventas, pagos, lista de usuarios y extremos de préstamos |
| RF-08 | Notificaciones | El sistema debe mostrar alertas de recuperación desde los 20 días de préstamo |
| RF-09 | Exportar a CSV | El sistema debe poder exportar la información de préstamos a formato CSV |
| RF-10 | Menú principal | El sistema debe mostrar un menú de consola visualmente amigable con todas las opciones disponibles |

### Requisitos No Funcionales

Los requisitos no funcionales definen **cómo debe comportarse** el sistema:

| ID | Categoría | Requisito |
|----|-----------|-----------|
| RNF-01 | **Usabilidad** | La interfaz de consola debe ser clara, con mensajes de error descriptivos y navegación intuitiva por menús numerados |
| RNF-02 | **Rendimiento** | Las operaciones de lectura y escritura de archivos deben completarse en menos de 2 segundos en condiciones normales |
| RNF-03 | **Fiabilidad** | El sistema debe validar toda entrada de usuario antes de procesarla, evitando datos corruptos en los archivos planos |
| RNF-04 | **Seguridad** | El módulo de administrador debe requerir autenticación con usuario y contraseña almacenados de forma segura |
| RNF-05 | **Mantenibilidad** | El código debe estar organizado con clases (`clsPrestamo`, `clsUsuarios`), documentado con el código `pf_Algoritmos` y seguir buenas prácticas de nombrado |
| RNF-06 | **Compatibilidad** | El sistema debe funcionar en Python 3.10 o superior en sistemas operativos Windows, Linux y macOS |
| RNF-07 | **Portabilidad** | Toda la información debe guardarse en archivos planos (.txt y .csv) sin necesidad de bases de datos externas |
| RNF-08 | **Escalabilidad** | La arquitectura orientada a objetos debe permitir agregar nuevas categorías de ítems o tipos de reportes sin refactorización mayor |
